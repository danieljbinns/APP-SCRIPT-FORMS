📋 ELI5 Implementation Plan
  1. Move FlowRenderer + StepModal JS into SharedComponents.html
    • Right now both Dashboard and RequestDetails each have their own copy of the workflow step renderer and modal open/close logic
    • Cut those two components out of Dashboard.html and move them into SharedComponents.html so any page can use them by just doing <?!= include('SharedComponents') ?>
    • Also move escapeHtml() and formatDate() — currently duplicated in both pages
  2. Move ALL modal/flow CSS into Styles.html
    • The step-flow row styles, modal overlay, step circle colours, connector lines — these are inline in Dashboard.html
    • Cut them to Styles.html. Nothing changes visually — Styles.html is already included on both pages
  3. Split DashboardHandler.js into DashboardDataHandler.js + DashboardUIHandler.js
    • DashboardDataHandler.js gets: getDashboardData(), getMyPendingTasks(), getWorkflowMapStats() — the 3 read-only data fetches
    • DashboardUIHandler.js gets: serveDashboard(), serveWorkflowMap() — the 2 page-serve functions
    • Nothing in the HTML pages changes yet — same function names, just different source files
  4. Extract RequestActionsHandler.js from RequestDetailsHandler.js
    • RequestActionsHandler.js gets: bumpRequest(), updateHireDate(), cancelRequest() — the 3 write/mutate operations
    • RequestDetailsHandler.js keeps: all the read-only detail fetchers + serveRequestDetails()
    • Both Dashboard and RequestDetails call these action functions — having them in their own file makes that shared responsibility explicit
  5. Rewrite RequestDetails.html to use shared FlowRenderer + StepModal
    • Remove the duplicate step-flow rendering code from RequestDetails.html
    • Call the shared FlowRenderer.render(steps, containerId) and StepModal.open(data) instead
    • This is the biggest client-side change — needs careful testing
  6. Add StepDetails.html as a dedicated drill-down page (future)
    • Currently getStepResultData() renders into a modal inside RequestDetails
    • New dedicated page: serveStepDetails(workflowId, stepType) → StepDetails.html
    • StepDetails calls getStepResultData() directly on load; uses shared components
    • Not doing this until Steps 1–5 are stable
  7. Promote to staging → prod after each step passes review
    • Each step above is one PR/commit, reviewed in dev before copying to staging then prod
    • Handler splits (steps 3+4) are zero-risk — server-side only, no HTML changes
    • Client-side changes (steps 1+2+5) need a visual diff check in staging first

📊 DashboardDataHandler.js — 3 functions
Purpose: Read-only data fetching for the Dashboard page and WorkflowMap. Never writes to sheets. Never serves HTML. No action logic. Safe to call without side effects.
getDashboardData()
What it does: Bulk-reads the pre-calculated Dashboard_View sheet (single API call), resolves the current user's access flags once, supplements any TERM_ rows with empType data, filters out Cancelled/Inactive, appends a termination fallback pass for pre-existing rows not yet in Dashboard_View, reverses the list (newest first), and returns JSON.
Returns: { success, workflows[], currentUser, canEditDates, isAdmin }
Called by: Dashboard.html on page load + refresh button
getMyPendingTasks()
What it does: Reads the Action Items sheet, filters to rows where the current user's email appears in the Assigned To column (comma-separated support), skips Closed items, looks up employee name from the Workflows sheet for each task, and returns the list.
Returns: { tasks: [{ tid, workflowId, category, taskName, employeeName, status, created }] }
Called by: Dashboard.html "My Tasks" panel on load
getWorkflowMapStats()
What it does: Reads the Workflows master sheet, skips Cancelled/Complete rows, groups active workflows by Current Step, splits into onboarding vs EOE buckets by Workflow ID prefix (TERM_ = EOE, CHANGE_ and EQUIP_ excluded).
Returns: { onboarding: { stepName: count }, eoe: { stepName: count } }
Called by: WorkflowMap.html live-status badges
🎨 DashboardUIHandler.js — 2 functions
Purpose: Serve HTML pages. These are the entry points called by the GAS Router. Tiny file — just template plumbing. No sheet reads, no business logic.
serveDashboard()
What it does: Creates template from 'Dashboard' file, injects baseUrl, spreadsheetId, and environment as template variables, evaluates and returns the HtmlOutput with title + viewport meta.
Called by: Router.js when URL param page=dashboard
serveWorkflowMap()
What it does: Same pattern — template from 'WorkflowMap', injects baseUrl, returns evaluated HtmlOutput.
Called by: Router.js when URL param page=workflowmap
🔍 RequestDetailsHandler.js — 6 functions
Purpose: All server-side logic for the RequestDetails page. Read-only data fetchers — never writes. Handles all 4 workflow types (Onboarding, EOE, Equipment, Status Change).
serveRequestDetails(workflowId)
Creates template from 'RequestDetails', injects baseUrl + workflowId, returns HtmlOutput. Called by Router.js when page=requestdetails&id=…
getRequestDetails(workflowId)
Main detail loader for Onboarding workflows. Reads Workflows sheet for header/status, then cross-references every specialist sheet (IT, HR, Safety, etc.) to build a complete checklist of completed/pending items. Returns structured object with employee info, step timeline, checklist, dates.
getTerminationDetails(workflowId)
EOE-specific detail loader. Reads the Terminations sheet + Workflows sheet to return term date, last day, manager, requested items, HR approval status, action item checklist.
getEquipmentRequestDetails(workflowId)
Equipment-request detail loader. Reads Equipment Requests sheet, returns requested items array, requester, assignee, approval status, delivery info.
getChangeDetails(workflowId)
Status Change detail loader. Reads Status Change Requests sheet, returns old/new position, site, effective date, approval chain status.
getStepResultData(workflowId, stepTarget)
Drill-down data for a specific completed step (IT Setup, HR Verification, Safety, etc.). Reads the step's specialist sheet to return the full submitted form data for display in the step modal. stepTarget is a string like 'it', 'hr', 'safety'.
⚡ RequestActionsHandler.js — 3 functions
Purpose: All write/mutate operations. These are the only handlers that modify sheet data. Used by BOTH Dashboard (quick actions from the modal) and RequestDetails (full detail view). Kept separate so write logic is never mixed into read-only data handlers.
bumpRequest(workflowId, targetStep)
What it does: Admin/HR-only. Moves a workflow forward (or backward) to a specific step. Validates access, updates the Workflows sheet status + current step, triggers the appropriate notification email, and re-syncs Dashboard_View.
Called by: Dashboard.html modal + RequestDetails.html bump button
updateHireDate(workflowId, newDateStr)
What it does: HR/IT/Admin only (canEditDates flag). Parses and validates the new date, writes it to the workflow row, re-syncs Dashboard_View so the card updates immediately.
Called by: RequestDetails.html hire date edit field only
cancelRequest(workflowId)
What it does: Admin only. Sets workflow status to Cancelled in the Workflows sheet and Dashboard_View, sends cancellation notification email to requester + manager.
Called by: Dashboard.html cancel button + RequestDetails.html cancel button
📦 Shared Files — what lives where
Rule: If a piece of code is used by more than one page — it lives in a shared file, not the page. Pages only contain logic that is specific to that page's view.
Styles.html
Contains ALL CSS. One <style> block. Included on every page via <?!= include('Styles') ?>. No CSS should live inline in any page HTML. This includes: base reset, layout, table styles, status pill colours, modal overlay/content, flow-step circles, connector lines, form field styles, theme class overrides (GeoCities, StarTrekTOS, etc.).
SharedComponents.html
Contains reusable JS components wrapped in <script> tags. Current: submitSpecialistTask(), buildDualList(), ThemeToggle. Moving in: FlowRenderer (renders the step-flow row from an array of step objects), StepModal (open/close/render the drill-down modal), escapeHtml(), formatDate().
SchemaConstants.js
Server-side only. Defines CONFIG (sheet IDs, names), SCHEMA (column index maps for every sheet), and ENVIRONMENT. Every other .js file reads from this — nothing hardcodes column numbers. Not included in HTML pages (server-side constant, not client-side).
