⚙ What Was Built

A gated multi-step workflow engine on Google Workspace — zero external servers, zero paid SaaS, zero database.

A fully custom HR workflow orchestration system built entirely on Google Apps Script + Google Sheets. The prior process was a series of Excel or Word forms emailed between departments — no tracking, no step enforcement, no audit trail, and no visibility on who had acted or when. Items fell through the gaps. Follow-up relied entirely on individual memory.

This system replaces that with a gated, multi-step request pipeline. Every step is locked until the prior one completes. Every specialist is notified and routed automatically the moment they're needed. Every action is timestamped, logged, and audited — with full change notifications dispatched any time submitted data is modified.

Workflow Types

  • New Hire
  • Termination / End of Employment (TERM + EOE)
  • Equipment Request
  • Status / Position Change

Cost avoidance — not hypothetical savings.
This was a solo, part-time effort — a full-time dev team wasn't available, and outsourcing was the realistic alternative. A senior contractor building an equivalent system would cost an estimated $65,000–$115,000 as a one-time build ($125–$175/hr over 500–650 hours), before ongoing maintenance. That's the direct cost avoided.

The broader picture: workflow automation platforms with comparable multi-step gating and routing — Kissflow, Process Street, Nintex — run $10–$25/user/month, or $180K–$450K+/year at TEAM Group's scale. Basic HR tools like BambooHR or Rippling cost $6–$9/user/month (~$108K–$162K/yr) but cover data collection only — no step enforcement, no specialist routing, no audit trail. This system runs on Google Workspace TEAM Group already pays for — zero incremental cost.

The most important gain is operational: tracked, enforced, accountable workflows replacing emailed spreadsheets with no visibility or follow-through mechanism.

🧪 QAT & Testing Evidence

6 deliberate QA sessions · 105 staging workflows · ~305 confirmed sheet writes — all 4 workflow types exercised.

All quality assurance and integration testing was performed by the developer personally — every form submission, every multi-step workflow walkthrough, and every email routing and edge-case verification was deliberate solo QA. Testing ran across three phases as the environment topology evolved.

Phase 1 — Prod-only (Jan 23 – Feb 24): No staging yet. Developer tested directly on prod. The ~72 NEW_EMP test workflows accumulated in the prod spreadsheet during this window.
Phase 2 — Staging as primary (Feb 24 – Apr 20): Staging was stood up once direct prod testing became unwieldy. At the cutover, prod was cleaned of test data — those ~72 NEW_EMP entries moved to staging. From Feb 24 onward prod received real HR submissions only. Staging became the active QA surface, adding 30 TERM/EOE + 3 CHANGE on top of the migrated NEW_EMP data.
Phase 3 — Dev for refactor (Apr 21 – Apr 29): Dev project (created Feb 24, kept dormant while staging was the active branch) activated for the Action Items refactor. Dev was seeded with ~244 NEW_EMP entries imported from prod for realistic reference data — those are not QA submissions. All dev testing was the developer running the new schema against all 4 workflow types during the Apr 21–29 sprint.

🏗 Architecture Highlights

Enterprise workflow capabilities — inside a Google Sheet.

The Action Items Refactor

The single largest architectural decision of the project. Six independent specialist result sheets were retired and replaced with a single normalized Action Items tab. Every task across every workflow type is now one row. Task lifecycle is uniform. Form data is stored as a JSON blob — schema changes don't require sheet migrations. Migration was handled by tools/migrate-prod-sheet/index.js — a local Node.js script with --dry-run / --execute modes. Zero changes to live GAS code during cutover.

Email Engine

~1,200 lines across EmailUtils.js and ChangeNotify.js. Dark-mode inline-styled HTML templates. Per-workflow subject builder with a hard 250-character cap (GAS MailApp.sendEmail throws Argument too large: subject beyond that — hit in production on Status Change workflows with long site names and classification strings). Change-audit diff tables show field / original value / updated value inline in the email body. Conditional routing: Safety team notified on SiteDocs/DSS-impacted changes, ID Setup notified on credential changes.

Sequential Gating Engine

Each workflow defines its steps as a config object. The engine evaluates current state, identifies the next pending step, generates a gated form URL, and refuses to advance if prior steps are incomplete. The same engine powers all 4 workflow types with zero duplication.

🔧 Key Technical Decisions

Every decision here solved a real constraint — version-controlled, maintainable, and safe to hand off.
DecisionRationale
.js over .gsEditors treat .js as JavaScript properly; cleaner clasp push/pull diffs
JSON blobs for form dataSchema-flexible — specialist forms can evolve without a sheet column migration
Script Properties for configEnvironment switching without code changes; no hardcoded spreadsheet or script IDs
Local migration toolingProd GAS untouched during data model migrations; safe dry-run workflow before execute
Inline styles in emailsGoogle Workspace strips <style> tags in some email clients; inline-only is safe
250-char subject capGAS throws Argument too large: subject above ~250 chars — hit in prod on Status Change
