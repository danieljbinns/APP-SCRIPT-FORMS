03Page Layout

One width. One card. Every form. All forms use .sim as the page wrapper, .cw for the content width, and .cf as the card surface. No narrow layouts, no centered small cards, no mixed widths.

04Context Banner

Every form has a context banner (.pctx) between the logo and the card. It tells the user which workflow this belongs to and any critical context they need before filling out the form. Use the blue variant for informational context, yellow (.pctx-w) for warning context (terminations etc).

Blue — Info Request & specialist forms
ℹ️  Workflow: NEW_EMP_20260422_001 · IT Setup is gated behind this step — complete before start date.
Yellow — Warning Termination forms
⚠️  Termination requests require HR approval before offboarding action items are created.
Content Rules
Form TypeBanner Content
New Employee RequestDescribes what submitting will trigger (which departments get action items)
Position / Site ChangeShows current employee snapshot: name · site · title · email
Termination RequestWarning: approval required, assets and access tracked — use yellow variant
All Approval FormsSubmitted by [name] on [date] · Workflow: [ID]
All Specialist FormsWorkflow: [ID] · any gating note (e.g. "IT Setup gated behind this step")

Approval and specialist forms use a form header block (.fhdr) inside the card as the first element — status badge, then h1 with employee name appended, then assignee / subtitle line. Request forms do not use this block (they start directly with a section heading).

08Input Fields

All inputs use class .fi with a label using .fl inside a .ff wrapper. Use .fr (auto-fit responsive grid) or .fr2 (fixed 2-col) to group related fields. Required fields get <span class="req">*</span> after the label.

09Radio Group

Use .rg (segmented control style) for mutually exclusive choices with 2–4 options. For Yes/No, always use .rg — never dropdowns. Selected state highlights in brand red.

10Checkbox Tiles

Multi-select options use a responsive tile grid (.cbt + .ct). Checked tiles highlight with red border and tint. Tiles with large sub-selections (Jonas, BOSS, Central Purchasing, Credit Card) trigger an expanding sub-section — see Standard 11.

Equipment Selection
11Expanding Sub-Section

When a checkbox tile represents a system or item that requires further sub-selection (too many options for tiles), a .sub-sec div expands inline beneath the tile grid when the parent checkbox is checked. Sub-selections use a compact .cbg grid. Used for: BOSS committees, Jonas job numbers, Central Purchasing sites, Credit Card types.

12Checklist Table

All step lists in all forms use the checklist table — no exceptions. Three columns always: Step · Status · Notes / Comments. No checkboxes, no bullet lists, no plain text lists. Every row starts with Pending (amber). Notes column is always present so specialists can log context on any step without needing another form.

16Status Badges

17Callout Boxes

Inline callout blocks (.co) for warnings or guidance inside a form section. Use .co-i for informational, .co-w for warnings. Distinguish from the context banner (.pctx) which is outside the card — callout boxes are inside the card, near relevant fields.

.co-i — Info
ℹ️ Pre-filled from the request — update only if the manager has changed.
.co-w — Warning
⚠️ Manager written approval is required before submitting a credit card application.
18Confirmation Screen

One confirmation layout for every single form. The form div is hidden and this div is shown in-place — same card, same width. Structure is always: ✅ icon · green title · short message · "Return to Dashboard" button · "You may close this window." Only the title and message text change per form.

The Standard — used identically on all 18 forms

20Request Forms — Structure Standard

The three request forms (New Employee, Position/Site Change, Termination) are the entry point for every workflow. They do not use the .fhdr form header block. They start directly with a section heading inside .cf.

Section Order
FormSections (in order)Ends with
New Employee RequestEmployee Information · System Access (+ sub-sections) · Equipment (+ sub-sections) · Additional Information.acts (Save Draft + Submit Request)
Position / Site ChangeChange Details · Additional Access Required · New Equipment Required · Additional Information.acts (Save Draft + Submit to HR)
Termination RequestEmployee Information · Assets to Retrieve · System Access to Revoke · Notes.acts (Save Draft + Submit to HR)
21Approval Forms — Structure Standard

Approval forms use the full form header (.fhdr with badge), show an info grid summary of the request, have a checklist for HR pre-submission tasks, then the HR Decision buttons, then a Notes field, then the submit button. HR Decision buttons always come after the checklist, before Notes.

Section Order
FormSections (in order)Ends with
Status Change Approval.fhdr · Request Summary (info grid) · HR Confirmation (editable fields) · Checklist · HR Decision · Notes.act1 — Submit Decision
Termination Approval.fhdr · Request Summary (info grid) · Checklist · HR Decision · Notes.act1 — Submit Decision
HR Verification & ADP.fhdr · Employee Details (info grid) · ADP Setup (input fields) · Checklist.act1 — Confirm & Complete
22Specialist Forms — Structure Standard

All specialist forms (IT Setup, Safety, Jonas, Fleetio, Business Cards, Credit Card, Central Purchasing, Employee ID, SiteDocs, Safety Offboarding) are rendered by a single sf() function that guarantees identical structure. The only variation is the employee info fields shown and the checklist steps. Forms that require additional inputs (30/60/90 Review, HR Verification) are built manually but follow the same component standards.

sf() Standard Structure (all specialist forms)
